01 · Idea del proyecto
Presenta: Oscar Morán

NUVIO: dos naturalezas de datos, un mismo sistema

Alcance acotado al contexto ecuatoriano (usuarios, ciudades, provincias) para mantener el modelo manejable sin perder complejidad relacional real.

Catálogo

Información de catálogo

Películas, series, temporadas, capítulos y reparto — del tipo que normalmente se extrae de fuentes externas (APIs de cine, bases públicas audiovisuales).

Transaccional

Información transaccional

Usuarios, perfiles, suscripciones e historial de visualización — propia de la operación diaria de un sistema de streaming.

Caso de uso planteado

NUVIO, una empresa ecuatoriana, licencia contenido audiovisual, lo cataloga en su plataforma y lo ofrece a usuarios registrados mediante planes de suscripción, permitiendo múltiples perfiles por cuenta, seguimiento de progreso de visualización y listas personalizadas.

02 · Modelo Entidad–Relación
Presenta: Oscar Morán

Cuatro bloques funcionales

El modelo agrupa las entidades del dominio antes de definir tipos de dato o llaves físicas.

Geografía

Región → Provincia → Ciudad, cadena jerárquica 1:N usada como ubicación de Persona.

Identidad

Persona como entidad base; User y Actor como especializaciones 1:1. User posee Perfil (1:N).

Catálogo

Estudio y Productora producen Película/Serie. Categoría clasifica (N:M). Serie → Temporada → Capítulo.

Consumo y negocio

User contrata Plan (Suscripción). Perfil visualiza contenido y guarda en MiLista.

02 · Modelo Entidad–Relación
Presenta: Oscar Morán

Decisiones explícitas del conceptual

Empresa y Logs

Se dejaron como entidades administrativas sin relación de negocio directa: Empresa es información institucional y Logs es auditoría transversal a todo el sistema.

Campos de auditoría

Usuario y fecha de creación/modificación no se modelan como entidad ni relación en el E/R conceptual — pertenecen al diseño físico, no al conceptual.

03 · Modelo físico
Presenta: Jim Vargas

Del E/R al esquema relacional

Reglas estándar de mapeo aplicadas para traducir el conceptual a tablas.

Llaves surrogate

Toda entidad se convirtió en tabla con id SERIAL, en lugar de identificadores conceptuales, simplificando FKs y evitando colisiones de nombre.

Relaciones 1:N

Resueltas con FK en la tabla del lado N. Ej: provincia.region_id, pelicula.estudio_id.

Relaciones N:M

Materializadas como tablas puente nuevas —no existían como entidades en el diagrama original—: pelicula_categoria, serie_categoria, reparto_pelicula, reparto_capitulo.

Entidades asociativas

Suscripción, Visualización y MiLista ya tenían atributos propios en el E/R — se mapearon directo como tabla, sin puente adicional.

Especialización 1:1

User y Actor como subtipos de Persona se resolvieron con FK única (persona_id UNIQUE) en lugar de fusionar atributos.

Bloque de auditoría

Cada tabla de negocio suma is_active, created_at, updated_at, created_by, updated_by (FK a user.id).

04 · Normalización
Presenta: Jim Vargas

Cumplimiento de formas normales

FormaVerificación
1FNTodos los atributos son atómicos; no hay grupos repetitivos ni columnas multivaluadas.
2FNLlaves surrogate en todas las tablas evitan dependencias parciales. En claves naturales compuestas (temporada, capítulo, reparto_pelicula, reparto_capitulo) los atributos dependen de la combinación completa que forma el UNIQUE.
3FNSin dependencias transitivas: Persona/User/Actor evita repetir nombre, cédula o email — separar estas tablas evita duplicar datos si una misma persona es usuario y actor a la vez.

Nomenclatura y auditoría

snake_case consistente (persona_id, created_at). Cada tabla de negocio incluye is_active, created_at, updated_at, created_by, updated_by, permitiendo rastrear autoría sin modelarlo en el conceptual.

Dato derivable almacenado

suscripcion.estado es derivable de fecha_vencimiento; se guarda por performance, asumiendo sincronización externa.

05 · Creación del script
Presenta: Matías Yépez

Ruptura de la dependencia circular

persona.created_by referencia user.id, pero user.persona_id referencia persona.id.

ddl_persona_user.sql
CREATE TABLE persona (
  id SERIAL PRIMARY KEY,
  nombres VARCHAR(100) NOT NULL,
  ...
  created_by INT,
  updated_by INT
);

CREATE TABLE "user" (
  id SERIAL PRIMARY KEY,
  persona_id INT NOT NULL UNIQUE REFERENCES persona(id),
  username VARCHAR(50) NOT NULL UNIQUE,
  password VARCHAR(255) NOT NULL,
  ...
  created_by INT REFERENCES "user"(id),
  updated_by INT REFERENCES "user"(id)
);

-- Se cierran las FKs de auditoría diferidas en persona
ALTER TABLE persona ADD CONSTRAINT fk_persona_created_by
  FOREIGN KEY (created_by) REFERENCES "user"(id);
ALTER TABLE persona ADD CONSTRAINT fk_persona_updated_by
  FOREIGN KEY (updated_by) REFERENCES "user"(id);

Presenta: Matías Yépez

Patrón polimórfico con CHECK

Evita una tabla intermedia genérica de tipo "contenido" para referenciar película o capítulo.

ddl_visualizacion.sql
CREATE TABLE visualizacion (
  id SERIAL PRIMARY KEY,
  perfil_id INT NOT NULL REFERENCES perfil(id),
  pelicula_id INT REFERENCES pelicula(id),
  capitulo_id INT REFERENCES capitulo(id),
  progreso DECIMAL(5,2),
  ...
  CHECK (pelicula_id IS NOT NULL OR capitulo_id IS NOT NULL)
);

05 · Creación del script
Presenta: Equipo NUVIO

Datos y consultas de prueba

Seed de datos

Geografía ecuatoriana, usuarios y perfiles, actores y contenido (películas y series con temporadas/capítulos), reparto, planes y suscripciones —incluyendo un caso vencido—, visualizaciones y listas.

12 consultas de prueba

Ejercitan joins multi-tabla, agregaciones, top-N y el patrón COALESCE necesario para las tablas polimórficas.

contenido_mas_visto.sql
SELECT COALESCE(p.titulo, cap.titulo) AS contenido,
       COUNT(*) AS veces_visto,
       ROUND(AVG(v.progreso), 2) AS progreso_promedio
FROM visualizacion v
LEFT JOIN pelicula p   ON p.id = v.pelicula_id
LEFT JOIN capitulo cap ON cap.id = v.capitulo_id
GROUP BY COALESCE(p.titulo, cap.titulo)
ORDER BY veces_visto DESC;
